Kapitel 1: Einleitung und Projektüberblick

1.1 Warum dieses Buch?

Clean Architecture ist kein „Framework“, sondern ein Denk- und Strukturierungsmodell, das Software langlebiger macht: besser testbar, einfacher wartbar und deutlich weniger abhängig von Infrastruktur-Details (Datenbank, Web-Framework, externe Services). In .NET lässt sich dieses Modell besonders gut mit CQRS (Command Query Responsibility Segregation) und MediatR kombinieren, um Use-Cases klar, modular und erweiterbar zu gestalten.

Dieses Buch erklärt Clean Architecture in .NET nicht abstrakt, sondern konsequent anhand eines durchgehenden Praxisprojekts.

1.2 Das Praxisprojekt: CleanArchitectureSolution

Als roter Faden dient das reale GitHub-Repository:

  • Repository: Kamal-Mohamed-Repos/CleanArchitectureSolution
  • Solution-Dateien im Root:
    • CleanArchitecture.sln
    • CleanArchitectureSolution.sln
  • Quellcode unter: src/

Die Lösung ist bereits in die klassischen Clean-Architecture-Schichten aufgeteilt:

  • src/CleanArchitecture.Domain
  • src/CleanArchitecture.Application
  • src/CleanArchitecture.Infrastructure
  • src/CleanArchitecture.WebAPI

Damit eignet sich das Projekt sehr gut, um Clean Architecture und CQRS nicht nur zu erklären, sondern direkt im echten Code wiederzufinden.

1.3 Zweck des Projekts

Das Projekt ist ein bewusst schlankes Beispiel für eine typische Geschäfts-Anwendung mit:

  • einer Domain (hier: u. a. „User“ als Entität),
  • Use-Cases in Form von Commands/Queries,
  • einer Persistenzschicht (EF Core),
  • einer REST API als Einstiegspunkt.

Als Minimal-Feature sind im Projekt bereits zwei Kern-Use-Cases umgesetzt, die wir im Buch immer wieder aufgreifen:

  1. Benutzer erstellen
  2. Benutzer abrufen (Liste)

Diese beiden Use-Cases reichen aus, um Clean Architecture, CQRS, MediatR, Dependency Injection, Persistenz und API-Flows vollständig und praxisnah zu erklären.

.NET / ASP.NET Core Web API

Die API ist ein ASP.NET-Core-Webprojekt (Microsoft.NET.Sdk.Web). Ziel-Framework:

  • net9.0 (siehe src/CleanArchitecture.WebAPI/CleanArchitecture.WebAPI.csproj)

Das ist wichtig, weil zum Bauen/Starten eine .NET-Version brauchst, die .NET 9 unterstützt.

CQRS ist im Projekt sichtbar durch die klare Trennung:

  • CleanArchitecture.Application/Commands
  • CleanArchitecture.Application/Queries

MediatR ist der „Dispatcher“, der die Requests an die passenden Handler weiterleitet. Das siehst du z. B. daran, dass der Controller nicht direkt Services/Repos aufruft, sondern _mediator.Send(...).

EF Core (Persistence)

Die Infrastruktur registriert einen EF-Core-DbContext und nutzt aktuell eine InMemory-Datenbank:

  • UseInMemoryDatabase("CleanArchDb") (in src/CleanArchitecture.Infrastructure/DependencyInjection.cs)

Damit ist das Projekt sofort startbar – ohne lokale SQL-Installation.

FluentValidation (+ Pipeline Behavior)

Validierung wird zentral über die MediatR-Pipeline angebunden:

  • Validatoren werden aus dem Application-Assembly registriert
  • ein ValidationBehavior<,> wird als PipelineBehavior registriert
    (siehe src/CleanArchitecture.Application/DependencyInjection.cs)

In der API gibt es zusätzlich einen globalen Exception Handler, der Validierungsfehler als HTTP 400 ausgibt (siehe src/CleanArchitecture.WebAPI/Program.cs).

1.7 Erste Orientierung: Wie ein Request durchs System läuft (Mini-Überblick)

Ein sehr typischer Flow im Projekt sieht so aus:

  1. HTTP Request trifft in UsersController ein
  2. Controller sendet einen Command/Query via MediatR
  3. MediatR findet den passenden Handler in der Application-Schicht
  4. Handler nutzt ein Interface (z. B. IUserRepository) aus Application
  5. Infrastructure liefert die konkrete Implementierung + EF Core DbContext
  6. Ergebnis geht zurück an Controller → HTTP Response

Beispiel: Benutzer erstellen (API → Command)

  • Controller ruft:
    • _mediator.Send(new CreateUserCommand(request.Name), ...)
  • Handler erstellt eine User-Entität und speichert über Repository.

Diese klare Trennung ist das Herzstück der Architektur – und wird in den nächsten Kapiteln systematisch auseinandergebaut und wieder zusammengesetzt.

Kapitel 2: Grundlagen der Softwarearchitektur

2.1 Was ist „Architektur“ in einer .NET-Anwendung?

Softwarearchitektur beschreibt die groben Bausteine eines Systems und ihre Zusammenarbeit:

  • Welche Schichten/Module gibt es?
  • Wer darf wen aufrufen?
  • Wo liegt Geschäftslogik?
  • Wie werden technische Details (Datenbank, HTTP, Logging) angebunden?
  • Wie bleiben Änderungen lokal, statt überall Seiteneffekte zu erzeugen?

Eine Architektur ist dann gut, wenn sie dich nicht bremst, sondern dich dabei unterstützt:

  • Features schneller und mit weniger Risiko zu liefern,
  • Änderungen isoliert vorzunehmen,
  • Tests gezielt zu schreiben,
  • Komplexität zu beherrschen.

Unser Praxisprojekt ist ein kleines, aber vollständiges Beispiel. Die grundlegenden Architekturprinzipien, die wir hier lernen, skalieren auf große Systeme.

2.2 Das Kernproblem: „Alles hängt an allem“

Ohne klare Struktur landen viele Projekte in einer Situation wie:

  • Controller enthält Logik
  • Controller spricht direkt mit DbContext
  • Validierung, Mapping und Business Rules verteilen sich „irgendwo“
  • Änderungen an der Datenbank schlagen bis in API/Frontend durch

Folgen:

  • geringe Testbarkeit (Integrationstests überall, Unit Tests schwierig)
  • hohe Kopplung (kleine Änderung → viele Dateien betroffen)
  • schlechte Verständlichkeit („wo passiert was?“)

Clean Architecture ist eine Antwort auf genau dieses Problem: Abhängigkeiten kontrollieren.

2.3 Grundbegriffe: Kopplung, Kohäsion, Abhängigkeiten

Kopplung (Coupling)

Wie stark hängt Modul A von Modul B ab?

  • Hohe Kopplung: Änderungen in B zwingen Änderungen in A
  • Niedrige Kopplung: A kann stabil bleiben, auch wenn B sich ändert

Kohäsion (Cohesion)

Wie „zusammengehörig“ sind Dinge innerhalb eines Moduls?

  • Hohe Kohäsion: ein Modul hat eine klare Verantwortung
  • Niedrige Kohäsion: „Müllschublade“-Klassen (Utility-Klassen, God-Services)

Abhängigkeiten

In der Praxis bedeutet das: Wer referenziert wen?

Im Beispielprojekt ist die Struktur schon so gewählt, dass:

  • CleanArchitecture.WebAPI die äußere Schicht ist (HTTP)
  • CleanArchitecture.Infrastructure technische Implementierungen enthält (EF Core, Repos)
  • CleanArchitecture.Application Use-Cases / CQRS enthält
  • CleanArchitecture.Domain die fachlichen Kernmodelle enthält

Und genau diese Abhängigkeitsrichtung ist der Punkt: Innen darf nicht von Außen abhängen.

2.4 SOLID (kurz) – die Prinzipien hinter „gut strukturierter“ Architektur

Clean Architecture ist stark von SOLID geprägt. Wir nutzen die Prinzipien nicht als Theorie, sondern als „Checkliste“, warum die Projektstruktur sinnvoll ist.

S — Single Responsibility Principle (SRP)

Eine Klasse/Komponente sollte genau eine Verantwortlichkeit haben.

Bezug zum Projekt:

  • UsersController macht nur HTTP/Transport (Requests/Responses).
  • CreateUserCommandHandler macht genau den Use-Case „User anlegen“.

O — Open/Closed Principle (OCP)

Erweiterbar ohne bestehendes Verhalten zu verändern.

Bezug:

  • Neue Use-Cases kommen als neue Commands/Queries + Handler hinzu.
  • Bestehende Handler müssen für neue Features meist nicht geändert werden.

L — Liskov Substitution Principle (LSP)

Subtypen müssen sich wie ihre Basistypen verhalten.

Bezug:

  • Wenn IUserRepository ein Verhalten verspricht, muss UserRepository es korrekt erfüllen.

I — Interface Segregation Principle (ISP)

Lieber mehrere kleine Interfaces statt ein großes.

Bezug:

  • IUserRepository ist (typischerweise) fachlich fokussiert, statt „IGeneralRepositoryEverything“.

D — Dependency Inversion Principle (DIP)

High-Level-Policy hängt nicht von Low-Level-Details ab.

Bezug (wichtigstes Clean-Architecture-Prinzip):

  • Application hängt von IUserRepository (Interface),
  • Infrastructure liefert UserRepository (Implementierung).

2.5 Schichtenarchitektur vs. Clean Architecture (wichtiger Unterschied)

Viele kennen klassische Layering-Ansätze:

  • Presentation → Business → Data

Das Problem: Häufig ist dabei „Business“ trotzdem abhängig von „Data“, weil z. B. EF-Core-Entities überall genutzt werden.

Clean Architecture dreht die Abhängigkeiten konsequent so, dass:

  • Domain und Application unabhängig bleiben
  • Infrastruktur ist austauschbar (SQL statt InMemory; REST statt Messaging; usw.)

Im Projekt siehst du das bereits an den Registrierungen:

  • In src/CleanArchitecture.WebAPI/Program.cs werden nur Layer registriert:
    • builder.Services.AddApplication();
    • builder.Services.AddInfrastructure();

... / CleanArchitectureSolution / src / CleanArchitecture.WebAPI / Program.cs

var builder = WebApplication.CreateBuilder(args);

// Clean Architecture layer registrations
builder.Services.AddApplication();
builder.Services.AddInfrastructure();

Das ist ein starkes Signal: Die API „komponiert“ das System, aber die Kernlogik liegt in Application/Domain.

2.6 Architekturziele, die wir im Projekt konkret erreichen wollen

Im Kontext des Projekts bedeutet „gute Architektur“ ganz konkret:

  1. Use-Cases sind eindeutig auffindbar
    Beispiel: „User erstellen“ liegt in CleanArchitecture.Application.Commands.

  2. Transport ist austauschbar
    Heute: REST Controller. Morgen: gRPC oder Messaging.
    Der Use-Case bleibt der gleiche.

  3. Persistenz ist austauschbar
    Heute: InMemory-Db. Morgen: SQL Server/PostgreSQL.
    Der Use-Case bleibt der gleiche.

  4. Validierung ist konsistent und zentral
    Nicht „mal hier, mal da“, sondern über Pipeline/Behavior.

  5. Testbarkeit
    Handler können ohne Webserver getestet werden.
    Controller sind dünn und brauchen kaum Logiktests.

Damit wir später präzise sprechen können, nutzen wir folgende Begriffe:

  • Entity (Domain): fachliches Objekt mit Identität (z. B. User)
  • Use-Case (Application): Anwendungsfall, der einen fachlichen Vorgang abbildet
    (z. B. CreateUserCommand / GetUsersQuery)
  • Command: schreibt Daten (ändert Zustand)
  • Query: liest Daten (ändert keinen Zustand)
  • Handler: implementiert den Use-Case
  • Infrastructure: technische Details (EF Core, Repositories)
  • WebAPI: HTTP-Einstieg, Routing, Statuscodes, OpenAPI

Kapitel 3: Clean Architecture verstehen

3.1 Die Grundidee: Abhängigkeiten nach innen

Clean Architecture (nach Robert C. Martin / „Uncle Bob“) lässt sich in einem Satz zusammenfassen:

Der fachliche Kern (Domain + Use-Cases) darf nicht von technischen Details abhängen.
Technische Details hängen umgekehrt vom Kern ab.

Das ist die zentrale Regel, die in unserem Beispielprojekt bereits sichtbar ist:

  • Die WebAPI registriert die Schichten, enthält aber keine Geschäftslogik.
  • Die Application enthält Commands/Queries/Handler (Use-Cases).
  • Die Infrastructure implementiert technische Details wie EF Core und Repositories.
  • Die Domain enthält die fachlichen Modelle (Entitäten).

3.2 Das „Onion/Ring“-Modell

So kannst du dir die Schichten vorstellen:

+---------------------------------------------------+
|                WebAPI (Delivery)                   |
|   Controller, HTTP, Statuscodes, OpenAPI/Swagger   |
+--------------------------|-------------------------+
                           v
+----------------------------------------------------+
|             Application (Use Cases)                |
|   CQRS: Commands/Queries, Handler, Interfaces      |
|   Validation Behavior (MediatR Pipeline)           |
+--------------------------|-------------------------+
                           v
+----------------------------------------------------+
|                 Domain (Core)                      |
|         Entities, Value Objects, Regeln            |
+--------------------------|-------------------------+
                           v
+----------------------------------------------------+
|            Infrastructure (Technical)              |
|      EF Core, DbContext, Repositories, IO          |
+----------------------------------------------------+

Wichtig: Die Pfeile zeigen nach innen, nicht nach außen.

  • Außen darf innen verwenden.
  • Innen darf außen nicht kennen.

3.3 Die Dependency Rule (Abhängigkeitsregel)

Die Dependency Rule ist der „Prüfstein“:

  • Domain kennt niemanden (oder nur sehr abstrakte Dinge).
  • Application kennt Domain.
  • Infrastructure kennt Application + Domain (weil sie die Interfaces implementiert).
  • WebAPI kennt Application + Infrastructure (weil sie alles „zusammensteckt“).

Im Projekt ist diese Richtung bereits in Program.cs erkennbar: Die API ruft nur AddApplication() und AddInfrastructure() auf.

3.6 Häufige Missverständnisse (und wie wir sie vermeiden)

  1. „Clean Architecture ist nur Ordnerstruktur“

  2. „Domain muss alles enthalten“
    Domain enthält den Kern. Aber „Use-Case-Abläufe“ gehören in Application.

  3. „Infrastructure ist böse“
    Infrastructure ist notwendig – aber sie darf nicht den Kern dominieren.

    Nein. CQRS trennt Lesen/Schreiben logisch. Event Sourcing ist optional.

Kapitel 5: CQRS – Theorie und Umsetzung im Projekt

CQRS (Command Query Responsibility Segregation) ist eines der wichtigsten Muster, um Use-Cases in Clean Architecture klar zu strukturieren. Unser Beispielprojekt setzt CQRS bewusst schlank um – ideal, um das Prinzip zu verstehen, bevor man es „industrialisiert“.

5.1 CQRS in einem Satz

Commands ändern Zustand, Queries lesen Zustand.
Und beide Wege werden getrennt modelliert.

Das heißt nicht automatisch:

  • zwei Datenbanken,
  • Event Sourcing,
  • komplexe Infrastruktur.

Es heißt zunächst nur: Trenne Schreiblogik und Leselogik sauber in Code und Modellierung.

5.2 Warum CQRS? (Das „WARUM“ hinter der Trennung)

In vielen Projekten passiert Folgendes:

  • Eine Methode „macht alles“: liest Daten, validiert, schreibt, berechnet, returned DTOs
  • Beim Wachsen des Systems wird jede Funktion „zu groß“
  • Man hat kein klares Vokabular für Use-Cases

CQRS schafft:

  1. Klare Intention

    • CreateUserCommand sagt sofort: „Schreiben“
    • GetUsersQuery sagt sofort: „Lesen“

  2. Saubere Verantwortlichkeiten

    • Command-Handler enthalten „Write Business Flow“
    • Query-Handler enthalten „Read Flow“ (oft schneller, optimierbarer)

  3. Bessere Testbarkeit

    • Command-Handler kann man isoliert testen (Input → Output + Side Effect)
    • Query-Handler kann man isoliert testen (Input → Output)

  4. Bessere Skalierung später

    • Wenn Lesen/Schreiben später unterschiedlich skaliert werden muss, ist der Code bereits getrennt.

5.3 Grundregeln: Command vs. Query

Command (Write)

Eigenschaften:

  • führt eine Änderung herbei (Create/Update/Delete, „Approve“, „Assign“, „Pay“…)
  • kann eine Id oder Status zurückgeben
  • sollte nicht „nebenbei“ massenhaft Daten lesen, außer zur Validierung/Entscheidung

Im Projekt:

  • CreateUserCommand : IRequest<Guid>

.../ CleanArchitectureSolution / src / CleanArchitecture.Application / Commands / CreateUserCommand.cs

public sealed record CreateUserCommand(string Name) : IRequest<Guid>;

Query (Read)

Eigenschaften:

  • ändert keinen Zustand
  • liefert Daten zurück (DTOs, ViewModels, ReadModels)
  • sollte keine Schreiboperation auslösen

Im Projekt:

  • GetUsersQuery : IRequest<IReadOnlyList<User>>

.../ CleanArchitectureSolution / src / CleanArchitecture.Application / Queries / GetUsersQuery.cs

public sealed record GetUsersQuery : IRequest<IReadOnlyList<User>>;

5.4 CQRS in der Ordnerstruktur (und warum das hilfreich ist)

Im Projekt ist CQRS auf Dateisystemebene sichtbar:

  • src/CleanArchitecture.Application/Commands/*
  • src/CleanArchitecture.Application/Queries/*

Das klingt simpel, ist aber didaktisch extrem stark:

  • Neue Entwickler:innen finden Use-Cases sofort.
  • Es gibt weniger Diskussionen „wo kommt das hin?“
  • Code Reviews werden einfacher, weil die Intention (Command/Query) schon im Pfad steht.

Kapitel 6: MediatR – Konzept und Verwendung im Projekt

MediatR ist in diesem Projekt der „Klebstoff“ zwischen WebAPI (HTTP) und Application (Use-Cases). Es sorgt dafür, dass Controller keine Use-Case-Implementierung kennen müssen – sie schicken nur eine Nachricht (Command/Query) und bekommen ein Ergebnis zurück.

6.1 MediatR kurz erklärt: „In-Process Messaging“

MediatR implementiert das Mediator-Pattern:

  • Ein Aufrufer (z. B. Controller) sendet einen Request: Send(request)
  • MediatR findet den passenden Handler: IRequestHandler<TRequest, TResponse>
  • Handler führt die Logik aus und liefert TResponse

Wichtig: Das ist In-Process, kein Message Broker.
Es ist eine strukturierte Methode, den Code modular aufzubauen.

6.2 Warum MediatR perfekt zu Clean Architecture passt

Clean Architecture will:

  • Use-Cases als eigenständige Einheiten
  • lose Kopplung zwischen „Delivery“ (HTTP) und „Use-Case-Ausführung“
  • Erweiterbarkeit ohne „God Service“

MediatR liefert genau das:

  1. Controller bleibt dünn
  2. Use-Cases sind klar auffindbar (Request + Handler)
  3. Cross-Cutting Concerns (Validierung, Logging, Performance) lassen sich als Pipeline einhängen

MediatR bietet Pipeline Behaviors (ähnlich Middleware, aber für Requests):

  • Request kommt rein
  • Behaviors laufen (z. B. Validierung)
  • Handler läuft

WARUM ist das elegant?

  • Validierung ist nicht in jedem Handler verteilt
  • du kannst neue Validatoren hinzufügen, ohne Handler zu ändern
  • Architektur bleibt sauber (Use-Case bleibt Use-Case)

  • versteckte Business Logik in HTTP-Schicht
  • schwer testbaren Code

8.1 Was ist die Domain in Clean Architecture?

Die Domain beantwortet die Fragen:

  • Welche Eigenschaften haben sie?
  • Welche Regeln/Invarianten gelten immer?

Wichtig: Domain ist nicht „Datenbankmodell“ und nicht „API-Contract“, sondern das fachliche Modell.

Im Projekt wird User aktuell in der Query direkt zurückgegeben:

  • GetUsersQuery : IRequest<IReadOnlyList<User>>
  • Controller gibt Ok(users) zurück

  • Entity (User): fachliches Modell + mögliche Regeln
  • DTO/ViewModel: Datenform für den Client (API)

Wenn du Entities direkt nach außen gibst, riskierst du später:

  • Kopplung zwischen Client und Domain
  • ungewollte Felder in der API (z. B. interne Properties)
  • schwierigere Versionierung der API

Wir nutzen den aktuellen Stand als Startpunkt und behandeln im Kapitel „Best Practices“ (Kapitel 13) die saubere Weiterentwicklung:

  • UserDto
  • Mapping/Projection in Queries
  • klare API-Contracts

8.4 Wo gehören fachliche Regeln hin? Domain vs. Application

Das ist eine der wichtigsten Entscheidungen in Clean Architecture.

8.4.1 Regeln, die „immer“ gelten → Domain

Beispiele:

  • „Name darf nie leer sein“
  • „Ein User muss eine gültige Id haben“
  • „Email muss eindeutig sein“ (je nach Modellierung)

Solche Regeln gehören idealerweise in die Domain, weil sie nicht davon abhängen sollten, ob die Daten über HTTP, Messaging oder Import kommen.

8.4.2 Regeln, die Use-Case-spezifisch sind → Application

Beispiele:

  • „Beim CreateUser darf Name max. 100 Zeichen haben“ (Policy des Use-Cases)
  • „Beim Import sind auch leere Namen erlaubt, aber werden als ‚Unknown‘ gesetzt“
  • „Beim UpdateUser gelten andere Regeln als beim CreateUser“

Solche Regeln sind oft besser in der Application-Schicht (z. B. via FluentValidation pro Command).

8.5 Was wir am Projekt konkret erkennen (und wie man es ausbauen würde)

Aktuell ist User ein anämisches Modell (nur Properties). Das ist für ein kleines Beispiel normal.

Ausbau-Idee (realistisch, aber kompatibel zur Architektur)

Wenn wir eine echte Domain-Regel einführen wollten, wäre eine typische Weiterentwicklung:

  • Name darf nicht leer sein
  • Name darf nicht nur Whitespace sein

Dann würde man oft:

  • Setter einschränken (z. B. private set)
  • Konstruktor/Factory erzwingen
  • Domain-Validierung zentralisieren

Beispiel (didaktisch, „realistisch“ — nicht 1:1 im Repo vorhanden):

C#

// Domain: User mit Invariante
public class User
{
    public Guid Id { get; private set; }
    public string Name { get; private set; }

    private User() { } // für EF o.ä., falls benötigt

    public User(Guid id, string name)
    {
        if (id == Guid.Empty) throw new 
                ArgumentException("Id must not be empty.", nameof(id));
        if (string.IsNullOrWhiteSpace(name)) throw new 
                ArgumentException("Name is required.", nameof(name));

        Id = id;
        Name = name.Trim();
    }
}

WAS passiert?

  • Objekt kann nicht in einen ungültigen Zustand geraten.

WARUM ist das wertvoll?

  • Domain schützt sich selbst.
  • Egal, aus welchem Use-Case/Adapter heraus User erstellt wird: die Regel gilt.

Im aktuellen Projekt wird die Validierung eher über die Application-Pipeline gedacht (FluentValidation + ValidationBehavior). Das ist ein legitimer Einstieg – später kann man Domain-Regeln ergänzen.

8.6 Domain Layer als „stabilster“ Teil des Systems

Clean Architecture behandelt Domain als das, was sich am seltensten ändern sollte.

  • Frameworks ändern sich (ASP.NET-Versionen, Hosting, OpenAPI-Libs)
  • Datenbanken ändern sich (InMemory → SQL → Cloud)
  • aber: „Was ist ein User?“ bleibt meist sehr stabil

Deshalb: Domain schlank, klar, regelorientiert.

Kapitel 13: Best Practices im Projekt

Dieses Kapitel ist bewusst praxisorientiert: Wir nehmen das bestehende Projekt als Grundlage und zeigen, wie man es „professioneller“ macht, ohne die Clean-Architecture-Idee zu verlieren.

Wichtig: Viele Punkte sind im Projekt bereits richtig angelegt (Schichten, CQRS, MediatR, Validation Pipeline). Best Practices bedeuten hier vor allem: saubere Struktur, bessere API-Formate, robustere Fehlerbehandlung, Testbarkeit und Weiterentwicklung.

13.1 Best Practice: Domain-Entities nicht direkt über die API zurückgeben

Status im Projekt

Aktuell liefert GET /api/users Domain-Entities:

... / CleanArchitectureSolution / src / CleanArchitecture.WebAPI / Controllers / UsersController.cs

var users = await _mediator.Send(new GetUsersQuery(), cancellationToken);
return Ok(users);

Und die Query ist:

... / CleanArchitectureSolution / src / CleanArchitecture.Application / Queries / GetUsersQuery.cs

public sealed record GetUsersQuery : IRequest<IReadOnlyList<User>>;

Problem

Wenn du User später änderst (z. B. neue Properties, interne Flags, Navigationsproperties), ändert sich automatisch dein API-Output.

Besser

  • Query liefert DTOs zurück, z. B. UserDto { Id, Name }
  • Domain bleibt intern

Typische Umsetzung:

  • GetUsersQuery : IRequest<IReadOnlyList<UserDto>>
  • Im QueryHandler Projektion statt Entities zurückzugeben

13.2 Best Practice: REST-Konventionen sauber machen (CreatedAtAction)

Im Projekt wird beim Erstellen aktuell zurückgegeben:

... / CleanArchitectureSolution / src / CleanArchitecture.WebAPI / Controllers / UsersController.cs

return CreatedAtAction(nameof(GetAll), new { id }, id);

Problem

GetAll hat keine Route, die {id} verwendet. In einer „sauberen“ REST-API würdest du eher:

  • POST /api/users → 201 Created + Location: /api/users/{id}
  • plus optional: Body mit { id: ... }

Besser

  • Endpoint hinzufügen: GET /api/users/{id}
  • dann CreatedAtAction(nameof(GetById), new { id }, ...)

Das ist kein Clean-Architecture-Thema, aber ein API-Qualitätsmerkmal.

13.3 Best Practice: Konsistente Fehlerformate (ProblemDetails) – guter Start, aber ausbauen

Was das Projekt schon gut macht

  • Validierungsfehler werden global zu HTTP 400 in ProblemDetails übersetzt (Program.cs).

Das ist ein sehr guter Ansatz: zentral, konsistent, verständlich.

Was man ausbauen kann

  • Nicht nur Detail als String, sondern strukturierte Fehler pro Feld (z. B. errors Dictionary).
  • Auch andere Exceptions zentral behandeln:
    • NotFound → 404
    • Unauthorized → 401
    • Forbidden → 403
    • unerwartete Fehler → 500 + Logging + TraceId

13.4 Best Practice: Result-Pattern statt „Exceptions als Flow“ (optional)

Aktuell:

  • Validation nutzt Exceptions (in .NET sehr üblich)

Optional (je nach Teamstil):

  • Result<T> / ErrorOr<T> / OneOf-Pattern, um Fehler als Daten zurückzugeben

Wann sinnvoll?

  • Wenn du viele fachliche Fehlerfälle hast („User already exists“, „Limit exceeded“)
  • Wenn du im Code explizite Fehlerpfade bevorzugst

Clean-Architecture-konform? Ja, solange:

  • Application keine HTTP-Statuscodes kennt
  • WebAPI die Übersetzung macht

13.5 Best Practice: Queries gezielt optimieren (Projektionen, Pagination)

Status im Projekt

GetAllAsync lädt alle User:

... / CleanArchitectureSolution / src / CleanArchitecture.Infrastructure / Persistence / UserRepository.cs

return await _context.Users.AsNoTracking()
                           .ToListAsync(cancellationToken);

Risiken bei Wachstum

  • Liste wird groß → Performance, Memory, Response-Größe
  • Keine Pagination

Besser

  • Query-Parameter: page, pageSize
  • Repo/QueryHandler: Skip/Take
  • Response: items + totalCount (oder nextCursor bei Cursor-Pagination)

13.6 Best Practice: Schreibmodelle und Lesemodelle trennen (CQRS „next level“)

Das Projekt benutzt aktuell dasselbe Modell (User) für Schreiben und Lesen.

Das ist okay, aber wenn Anforderungen steigen:

  • Write Model: Domain-Entity (reich, regelgetrieben)
  • Read Model: DTOs/ViewModels (für UI/API optimiert)

Das kann bedeuten:

  • Queries nutzen Projektion (Select) auf DTO
  • Commands arbeiten mit Domain-Entity und Repositories

13.7 Best Practice: Unit Tests auf Handler-Ebene (sehr gut geeignet)

Weil die Controller dünn sind und Handler über Interfaces arbeiten, ist Testing sehr einfach:

Was testest du typischerweise?

  • CreateUserCommandHandler:

    • erzeugt User mit Name
    • ruft Repository einmal auf
    • gibt Id zurück

  • CreateUserCommandValidator:

    • leerer Name → Validation Fehler
    • Name > 100 → Validation Fehler

  • GetUsersQueryHandler:

    • ruft GetAllAsync
    • gibt Ergebnis zurück

WARUM ist das so gut?

  • Keine Webserver-Startkosten
  • Kein EF Core nötig (Repo mocken)
  • Schnelle Tests → schnelleres Feedback

13.8 Best Practice: Logging/Tracing an der richtigen Stelle

Ein häufiges Setup:

  • Logging als MediatR Pipeline Behavior (z. B. „RequestName + Duration“)
  • CorrelationId/TraceId in Responses (ASP.NET Core kann das über Activity/OpenTelemetry)
  • Fehlerlogging zentral im Exception Handler/Middleware

Das hält Handler sauber, liefert aber dennoch Observability.

13.9 Best Practice: Schichtenabhängigkeiten regelmäßig prüfen

In .NET kann man versehentliche Abhängigkeiten schnell einschleppen (z. B. Domain referenziert EF Core). Best Practices:

  • Projekt-Referenzen minimal halten
  • Architekturtests (z. B. NetArchTest) einsetzen
  • Code Reviews mit Fokus: „Abhängigkeiten nach innen“

Im Projekt ist die Grundstruktur gut, und genau das sollte man schützen.

13.10 Best Practice: InMemory als Dev-Start – später echte DB + Migrationen

Das Projekt nutzt:

  • UseInMemoryDatabase("CleanArchDb")

Das ist top für den Start. Für Produktion würdest du üblicherweise:

  • Provider wechseln (z. B. SQL Server/PostgreSQL)
  • ConnectionStrings per Config
  • Migrationen
  • Seeding optional

Das bleibt clean, wenn es in Infrastructure bleibt.

Kapitel 14: Erweiterungen und nächste Schritte

Dieses Kapitel ist die praktische Roadmap: Wenn du aus CleanArchitectureSolution ein robustes, produktionsnahes Projekt machen willst – welche Schritte sind sinnvoll, in welcher Reihenfolge, und warum?

Wir bleiben bewusst beim bestehenden Projektaufbau (Domain/Application/Infrastructure/WebAPI) und erweitern entlang der gleichen Prinzipien.

14.1 Roadmap: Vom Lernprojekt zur produktionsreifen Basis

Phase 1 (kurzfristig, hoher Nutzen)

  1. DTOs für API Responses (Read Models)
  2. GET /api/users/{id} (für korrektes CreatedAtAction)
  3. Pagination für GET /api/users
  4. Sauberer Fehlerstruktur (ProblemDetails mit Feldfehlern)

Phase 2 (mittelfristig)

  1. Persistenz auf echte DB umstellen (SQL Server/PostgreSQL) + Migrationen
  2. Testsuite aufbauen (Unit Tests für Handler/Validatoren, Integrationstests für API)
  3. Logging/Telemetry (MediatR Logging Behavior + OpenTelemetry)

Phase 3 (langfristig / Skalierung)

  1. AuthN/AuthZ (JWT, Policies/Rollen)
  2. Read/Write Modelle stärker trennen (Projektionen, eventuell separate Read DB)
  3. Outbox/Eventing (falls du Events brauchst)
  4. CI/CD (GitHub Actions, Quality Gates)

14.2 Konkrete Erweiterung 1: GET /api/users/{id} ergänzen

Warum?

  • REST-konformer: Ressourcen sind über Id adressierbar.
  • POST kann korrekt Location auf die neue Ressource setzen.
  • Viele Clients brauchen „Details laden“ nach Create.

Wo kommt was hin?

  • WebAPI: neuer Endpoint im UsersController
  • Application: GetUserByIdQuery + Handler
  • Infrastructure: IUserRepository.GetByIdAsync(...) + Implementierung

WAS passiert?

  • Du führst einen neuen Read-Use-Case ein, ohne existierende Use-Cases kaputt zu machen.

WARUM ist das Clean?

  • Erweiterung durch neue Use-Cases (OCP), keine „God Service“-Änderungen.

14.3 Konkrete Erweiterung 2: DTOs einführen (API-Contracts stabilisieren)

Ziel

  • Domain bleibt intern.
  • API gibt nur das zurück, was der Client braucht.

Typischer Ansatz

  • UserDto (z. B. Id, Name)
  • QueryHandler projiziert auf DTO

Warum in Queries besonders sinnvoll?

  • Queries sind read-optimiert.
  • Projizieren spart oft Performance und reduziert Kopplung.

14.4 Konkrete Erweiterung 3: Pagination für GET /api/users

Warum?

„Alle User“ funktioniert nur bei kleinen Datenmengen. Pagination ist Standard.

API-Design (Beispiel)

  • GET /api/users?page=1&pageSize=20
  • Response: items + totalCount (oder Cursor)

Schichtzuordnung

  • WebAPI: Query-Parameter in Request/DTO
  • Application: GetUsersPagedQuery
  • Infrastructure: Repo-/Query-Implementierung mit Skip/Take

14.5 Konkrete Erweiterung 4: Persistenz von InMemory → echte DB

Aktuell:

  • EF Core InMemory
  • sehr gut für lokale Entwicklung

Nächster Schritt in Infrastructure:

  • Provider wechseln (z. B. Microsoft.EntityFrameworkCore.SqlServer / Npgsql.EntityFrameworkCore.PostgreSQL)
  • Connection String in appsettings.json
  • DbContext bleibt derselbe, Mappings bleiben nahezu gleich
  • Migrationen (dotnet ef migrations add ...)

Warum ist das Clean Architecture-konform?

  • Application/Domain bleiben unverändert.
  • Nur Infrastructure + Config in WebAPI ändern sich.

14.6 Konkrete Erweiterung 5: Tests strategisch aufbauen

1) Unit Tests (schnell, viele)

  • Validatoren (CreateUserCommandValidator)
  • Handler (CreateUserCommandHandler, GetUsersQueryHandler)

2) Integrationstests (weniger, aber wertvoll)

  • API startet in Testhost
  • echte Pipeline (MediatR + Validation + Exception Handling)
  • prüft Statuscodes und Responseformate

Warum ist das in diesem Projekt besonders gut machbar?

  • Use-Cases sind isoliert.
  • Infrastruktur ist austauschbar (für Tests weiterhin InMemory).

14.7 Konkrete Erweiterung 6: Observability (Logging/Tracing/Metrics)

Empfohlen:

  • MediatR Behavior „Performance/Logging“
    • logge Request-Name + Dauer
  • OpenTelemetry in WebAPI
    • Traces über HTTP Requests hinweg
    • optional DB-Calls

Wichtig: Logging gehört nicht in jeden Handler einzeln, sondern zentral (Behavior/Middleware).

14.8 Konkrete Erweiterung 7: Sicherheit (AuthN/AuthZ)

Sobald echte Systeme entstehen, ist Auth fast immer nötig:

  • JWT Bearer Authentication
  • Policies (z. B. „Admin“ darf User löschen)
  • Claims-basiert

Saubere Zuordnung:

  • WebAPI: Authentication/Authorization Middleware, [Authorize] Attribute
  • Application: Entscheidungen als Use-Cases („DeleteUser“), idealerweise unabhängig von HTTP
  • Infrastructure: ggf. Identity Provider Integration

14.9 Langfristiges CQRS: getrennte Read Models / Projektionen

Wenn du sehr viele Reads hast:

  • Queries liefern optimierte DTOs
  • du kannst sogar auf separate Read-Datenhaltung gehen (nicht zwingend!)

Wichtig: CQRS muss nicht „Big Bang“ sein. Es kann stufenweise wachsen:

  • Erst: ordentliche Trennung in Code (wie im Projekt)
  • Dann: DTOs + Projektionen
  • Dann: Performanceoptimierungen
  • Dann: ggf. getrennte Modelle/Stores
